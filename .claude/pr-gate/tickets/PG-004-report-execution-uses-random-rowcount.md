## Summary

`POST /api/reports/:id/execute` inserts a real database record with `rowCount` and `durationMs` generated by `Math.random()`. These random values are persisted to the `report_executions` table and returned as `data` in the API response. Callers have no way to distinguish a real execution result from a fabricated one without checking the `mock: true` field, which is absent from this route (unlike `/api/query/execute` which includes `mock: true`).

## Evidence

- `src/server/routes/reports.ts:172-174`:
  ```ts
  const rowCount = Math.floor(Math.random() * 1000) + 1;
  const durationMs = Math.floor(Math.random() * 2000) + 50;
  ```
- `src/server/routes/reports.ts:180` — `status: "mock"` is stored, but the response body does not include a top-level `mock: true` flag.
- The test at `tests/server/reports.test.ts:83-85` asserts `body.data.status === "mock"` — the status sentinel is present, but callers that do not inspect the status field would see a plausible-looking rowCount (1–1000) and durationMs (50–2049).

## Why this matters

Persisting random execution history pollutes the `report_executions` table with data that looks real. Historical trend analysis, dashboards showing "last N executions", and any future feature that queries execution history will operate on noise. The inconsistency between this endpoint (no `mock: true` in response) and `/api/query/execute` (which includes `mock: true`) also creates an inconsistent API contract.

## Proposed fix

1. Return a consistent `mock: true` field at the top level of the response alongside `data`.
2. Use deterministic stub values (e.g., `rowCount: 0, durationMs: 0`) instead of `Math.random()`, or document clearly that the values are random.
3. Alternatively, do not insert into `report_executions` when in stub/mock mode, or mark the record clearly with `status: "mock"` and filter it out of execution history queries.

## Acceptance checks

- [ ] `POST /reports/:id/execute` response includes `mock: true` at the top level.
- [ ] Tests assert the `mock` flag in the response, not just the status field.
- [ ] Execution history endpoint either excludes mock records or clearly labels them.
